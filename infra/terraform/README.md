# AWS free-tier infrastructure

Terraform stack that provisions everything the GitHub Actions deploy workflow needs:

| Resource | Purpose | Free-tier note |
| --- | --- | --- |
| EC2 `t2.micro` | API + worker + beat + Redis (all containerised) | 750 hrs/mo for 12 months |
| RDS `db.t3.micro` (Postgres 16) | Application database, PostGIS extension | 750 hrs/mo + 20 GB storage + 20 GB backup |
| S3 bucket | Trained joblib artefacts (versioned, encrypted) | 5 GB free / 12 mo |
| ECR (two repos) | `transport-delay-predictor/api`, `transport-delay-predictor/worker` | 500 MB private storage / 12 mo (lifecycle policy keeps last 5) |
| IAM OIDC provider + role | GitHub Actions assumes via short-lived tokens — no AWS keys in GitHub | Free |
| EC2 instance profile | ECR pull + S3 r/w + SSM-managed (Session Manager, no port 22) | Free |
| SSM Parameter Store | DB password, connection URLs, model bucket name | Free (standard params) |

## Prerequisites

- An AWS account with the 12-month free tier still active
- Terraform ≥ 1.6
- AWS CLI configured with credentials that can create IAM/VPC/RDS/EC2 resources

## Apply

```bash
cd infra/terraform
terraform init
terraform plan -out tfplan
terraform apply tfplan
```

The first `apply` takes ~10 minutes (RDS provisioning is the slow part).

## Wire the GitHub repo

After `apply` succeeds:

```bash
terraform output github_actions_role_arn
terraform output ec2_instance_id
```

In the GitHub repo settings → **Secrets and variables → Actions**, create:

| Type | Name | Value |
| --- | --- | --- |
| Secret | `AWS_DEPLOY_ROLE_ARN` | `terraform output -raw github_actions_role_arn` |
| Secret | `AWS_EC2_INSTANCE_ID` | `terraform output -raw ec2_instance_id` |
| Variable (optional) | `AWS_REGION` | Defaults to `us-east-1` if unset |

That's it — pushing to `main` will now build images, push to ECR, and trigger `infra/scripts/deploy.sh` on the EC2 host via SSM.

## What `terraform apply` does, in order

1. **VPC lookup** — uses the default VPC + its subnets. We don't provision a NAT gateway (~$33/mo) because the EC2 host lives in a public subnet with a public IP.
2. **Security groups** — EC2 accepts 80/8000 from the world; RDS only accepts 5432 from the EC2 SG.
3. **RDS** — db.t3.micro Postgres 16, single-AZ, 20 GB gp2, encrypted, master password generated by the `random_password` provider and stored as an SSM SecureString.
4. **S3 + ECR** — model artefact bucket (versioned, encrypted, public access fully blocked) + two ECR repos with a 5-image lifecycle policy.
5. **IAM** — instance profile + OIDC role + scoped policies.
6. **EC2** — t2.micro on Amazon Linux 2023, IMDSv2-only, encrypted gp3 root volume. The user-data script installs Docker + the Compose plugin, clones the repo, renders `.env` from SSM, runs the PostGIS `CREATE EXTENSION` against RDS, and starts the compose stack (the first time, no images exist yet — that's fine, the first GH Actions deploy will pull them).

## Tearing it all down

```bash
terraform destroy
```

S3 versioning means the bucket might refuse to delete while versioned objects exist — empty it first (`aws s3 rm --recursive s3://<bucket> --include "*"` followed by `aws s3api delete-objects --bucket <bucket> --delete "$(aws s3api list-object-versions ...)"`).

## Variables worth knowing

See `variables.tf` for the full list. Common overrides:

```bash
terraform apply \
  -var aws_region=ap-southeast-2 \
  -var ec2_instance_type=t3.micro \
  -var github_repository=my-org/my-fork
```

## Free-tier honesty

This stack stays in free tier **only if**:
- The account is within its 12-month introductory window.
- You don't enable Multi-AZ on RDS (off by default).
- Egress stays under 100 GB/month (web traffic from EC2 to clients).
- ECR storage stays under 500 MB total — the 5-image lifecycle policy is the main lever; a Python+ML image runs ~600 MB compressed, so even 1 retained image can push you over. Watch `aws ecr describe-image-scan-findings` after a few deploys.
